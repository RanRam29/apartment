# 🗄️ מודל נתונים וסכמות (Data Models & ERD)

המערכת משתמשת בגישת **Polyglot Persistence**, התאמת הכלי הנכון לסוג המידע.

## 1. PostgreSQL (מידע רלציוני עסקאות וישויות ליבה)
כאן אנו מנהלים את האובייקטים הדורשים שלמות נתונים (ACID) וקשרים חזקים:
* **`Users`**: מזהה, אימייל, סיסמה מוצפנת, תפקיד (`Role: TENANT | LANDLORD`), תאריך הרשמה.
* **`Apartments`**: פרטי הנכס (כתובת, מחיר, חדרים), קשור ב-Foreign Key (FK) ל-`Users` (המשכיר).
* **`Swipes`**: טבלת תיעוד החלקות. עמודות: `tenant_id`, `apartment_id`, `direction` (Right/Left), `timestamp`.
* **`Matches`**: נוצרת כאשר שוכר עשה לייק לדירה, והמשכיר אישר את הליד. קושרת `tenant_id` עם `apartment_id`.

## 2. MongoDB (מידע מסמכי, גמיש והיסטורי)
מושלם למידע עם סכמה משתנה או נפחים עצומים שדורשים חיפושים מהירים על טקסט:
* **`Messages`**: שמירת היסטוריית הצ'אט. כל מסמך מכיל `match_id`, `sender_id`, `content`, `timestamp`. מאונדקס לפי `match_id` לשליפה מיידית של שיחות.
* **`UserPreferences`**: טווח מחירים, אזורים, מילות מפתח, פילטרים ייחודיים (חיות מחמד, מעלית). נשמר כ-JSON מקונן וגמיש.

## 3. Redis (ניהול זמני ו-Caching)
* **`Feed Cache`**: אילו דירות המשתמש כבר ראה? נשמר בזיכרון המהיר כדי שה-`SwipeScreen` לא ייאלץ להמתין למסד הנתונים בכל החלקה.
* **`Sessions & NLP Cache`**: שמירת תוצאות של פענוח חיפושים מה-Gemini AI למשך 24 שעות כדי לחסוך קריאות כפולות (וקריאות API יקרות).
